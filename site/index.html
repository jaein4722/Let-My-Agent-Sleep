The problem

Waiting is the expensive part.

A long job doesn't need a babysitter. But most agents don't know that — so they sit there and spend.

Without LMAS

Polling every 10 seconds for 12 hours

The agent kicks off a training run, then loops: read the log, wait, read the log, wait. Every check spends tokens and context on a job that hasn't changed. Multiply by a 12-hour run and the bill is all overhead.

≈ 4,320 log checks · 0 useful work between them
With LMAS

One handoff, one wake-up

The agent starts the job and immediately receives LMAS_HANDOFF v1, so it stops. An independent tmux watcher owns the job. When it finishes, the session is woken once with the completion event and full context.

1 handoff · 1 wake-up · the agent designs the next run
How it works

Three moving parts. No daemon.

All state lives in plain files under .lmas/runs/<run_id>/ — stdout, stderr, handoff, completion event, exit code. Nothing to run, nothing to host.

STEP 01

Handoff

The agent starts the job through LMAS and gets LMAS_HANDOFF v1 back immediately — run_id, STARTED, resume instruction. It stops. No polling, no "continue" spam.

STEP 02

Independent tmux watcher

The job runs under a tmux watcher outside the agent process. It captures logs and exit code, then writes LMAS_COMPLETION_EVENT v1 when the job ends — success, failure, or cancelled.

STEP 03

Session re-entry

An adapter injects the completion event back into the original session. The agent wakes with full context, inspects stdout.log, summarizes results, and continues the work.

Durability

Survives what native background tasks don't.

The watcher and the wake-up live outside the agent app. Native background-task features die with the app — LMAS keeps running and still wakes the session.

The agent app is closed Quit the frontend, close the window — the tmux watcher keeps owning the job.

RUNNING SUCCEEDED FAILED CANCELLED LOST

LOST means the watcher died without a completion event — a machine reboot, say. LMAS reports it honestly and never silently relaunches your job.

Adapters

Three agents, three honest tiers.

Support is real where we say it's real. Here's exactly how each session gets woken back up.

OpenCode

Primary

Fully supported. The plugin provides lmas_start, lmas_status, and lmas_cancel tools. Completion is POSTed to the persistent opencode serve HTTP API for real-time session wake-up.

→ real-time HTTP wake-up

Codex

Supported

Supported through the installed skill. When the job completes, the original session is resumed via codex exec resume so the agent can pick the work back up.

→ resume via codex exec resume

Claude Code

Experimental

Experimental via claude --resume. Appending the completion to the session transcript is reliable; an already-open window shows it after a restart. Automatic resume isn't guaranteed on every frontend.

→ transcript append via claude --resume
Design principles

What it deliberately won't do.

The non-goals are the point. They're what make it safe to hand a multi-day job to an agent.

No timeouts

Your job is never killed by a clock. Runs may go 3+ days. Diagnosing and terminating a hung job belongs to you — never to the skill.

Not another scheduler

No built-in job chaining. After each completion the agent designs the next experiment and starts the next run. LMAS just carries the handoff and the wake-up.

Zero infrastructure

bash + tmux + plain files. No daemon, no cloud, no database. If you can run tmux, you can run LMAS — and read every artifact with cat.
